# NKS-CTX-000001 — Operating Model AI Context Pack

## Purpose

Provide durable AI context for future reasoning about the Nelson Knowledge System without re-deriving the architecture from conversation history.

## Core Identity

The Nelson Knowledge System is a knowledge manufacturing system.

It is not a publishing calendar, content machine, note archive, or media workflow.

Publishing is a downstream output.

The objective is to manufacture reusable intellectual property from source material.

## Mission

Convert fragmented source material into proof-bounded, relationship-aware, visually legible, reusable knowledge assets.

## Source Material

Source material may include:

- conversations,
- whitepapers,
- executive profiles,
- resumes,
- role analyses,
- diagrams,
- proof records,
- feedback,
- research notes,
- future telemetry.

## Manufacturing Pipeline

```text
Source
→ Evidence / Proof
→ Canonical Artifact
→ Narrative Arc
→ Visual Knowledge Package
→ Output Package
→ Distribution / Use
→ Feedback
→ Corpus Enrichment
```

## Operating Priorities

1. Capability growth.
2. Corpus coherence.
3. Proof strength.
4. Relationship density.
5. Reuse multiplication.
6. Automation leverage.
7. Output readiness.

## Anti-Regression Rule

Never optimize for publication count when capability growth is available.

If publication and capability conflict, capability wins.

## Core Subsystems

| Subsystem | Function |
|---|---|
| Source Intake | Identifies and classifies raw material. |
| Corpus Manufacturing | Converts durable ideas into canonical artifacts. |
| Proof System | Defines what can and cannot be claimed. |
| Narrative Arc | Converts artifacts into comprehensible journeys. |
| Visual Knowledge | Makes relationships and systems legible. |
| Knowledge Graph | Links sources, artifacts, proof, visuals, outputs, opportunities, and feedback. |
| Coherence Engine | Scores whether an artifact strengthens the system. |
| Opportunity Engine | Identifies what each artifact can become. |
| Contract Layer | Preserves vendor independence and adapter replaceability. |
| Execution Policy | Selects next work based on capability growth. |

## Current Key Rules

- Do not ingest everything; ingest what compounds.
- Do not publish because something is ready; publish only when release advances the system.
- Do not treat Drive, GitHub, Medium, or any vendor as the system.
- Own contracts and interfaces; buy or replace execution surfaces.
- Every public output must preserve proof boundaries.
- Every canonical artifact must link to source.
- Every high-value artifact should be evaluated for multiple downstream uses.

## Current High-Value Clusters

1. Foundational NKS Operating Model.
2. Executive Value / Clarity / Coherence.
3. Agentic AI Governance.
4. Visual Knowledge System.
5. Integration and Adapter Architecture.
6. Career / Opportunity Engine.

## Execution Behavior

When instructed to execute:

1. Load current state.
2. Select the highest-priority unblocked capability-building task.
3. Create or update concrete repository artifacts.
4. Verify the result.
5. Update state or create an execution snapshot.
6. Continue until a genuine blocker or execution-window boundary.

## Genuine Blockers

- Required user approval for public release.
- Missing source material for a claim.
- Tool write failure that cannot be bypassed.
- Missing external credentials for third-party publication.
- Safety or privacy boundary.

## Non-Blockers

- Drive intermittency when GitHub can preserve state.
- Lack of publication connector when manual/adapter path can be modeled.
- Incomplete external distribution when internal capability work remains.
- Absence of perfect visuals when SVG/brief artifacts can advance the visual system.
