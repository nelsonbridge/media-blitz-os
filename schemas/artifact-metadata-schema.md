# Artifact Metadata Schema

Every durable content asset should be tracked with the following metadata.

## Core Fields

| Field | Description |
|---|---|
| Artifact ID | Stable identifier, e.g. `MB-0001`. |
| Title | Working or final title. |
| Artifact Type | Article, template, whitepaper, post, carousel, framework, brief, etc. |
| Pillar | Primary content pillar. |
| Subpillar | More specific thematic category. |
| Source | Conversation, uploaded file, Drive file, resume, whitepaper, role analysis, etc. |
| Source Location | Link, file path, chat reference, or note. |
| Primary Audience | Main intended reader. |
| Secondary Audience | Additional useful reader group. |
| Career Alignment | Role/opportunity type the asset supports. |
| Strategic Value | Why the asset matters. |
| Authority Value | How it demonstrates expertise. |
| Evergreen Rating | High, Medium, Low. |
| Time Sensitivity | Evergreen, Seasonal, Current, Reactive. |
| Evidence Requirement | None, Light, Moderate, Heavy. |
| Evidence Status | Not Needed, Needed, Verified, Incomplete. |
| Risk Level | Low, Medium, High. |
| Brand Drag Risk | Low, Medium, High. |
| Status | Planned, Drafted, Review, Published, Archived, Blocked. |
| Canonical Location | Drive URL, GitHub path, Medium URL, etc. |
| Derivatives Required | LinkedIn, X, Facebook, Instagram, Pinterest, PDF, etc. |
| Related Assets | IDs of related artifacts. |
| Last Reviewed | Date reviewed. |
| Next Review | Date for update/relevance review. |

## Rule

No asset should be published publicly without enough metadata to understand its audience, strategic value, evidence posture, and career relevance.
