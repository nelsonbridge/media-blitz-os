# NKS-STD-003 — Capability Hierarchy

Status: Approved  
Version: 1.0

## Purpose

Define how foundational capabilities, architectural domains, programs, implementations, and features relate without embedding commercial product structures in code.

## Hierarchy

1. **ME capability class** — foundational civilization-scale capability concept.
2. **Deity domain** — enduring architectural capability family.
3. **King program** — governed implementation, initiative, release train, or campaign.
4. **Component** — executable or record-level unit within a domain or program.
5. **Implementation** — current human, AI, service, adapter, or automation realizing a component.

## Parentage

Every canonical capability except a root capability must declare a parent.

Every program must declare one or more governing deity domains.

Every component must declare:

- the capability it provides;
- the capability or program that owns it;
- the capabilities it requires;
- its authority class;
- its license eligibility.

## Overlap Rule

When a deity and a king both align semantically:

- use the deity for the durable domain;
- use the king for a bounded realization of that domain.

Example:

- `ANU` remains the enduring origin-authority domain.
- a future named king program may implement a constitutional migration, release, or governance initiative under ANU.

## Current Root Assignments

### ANU

Type: Deity domain  
Purpose: Origin Authority  
Hierarchy: Root constitutional capability

### ENKI

Type: Deity domain  
Purpose: Knowledge Engine  
Parent: ANU

ENKI receives strategic intent from ANU and converts it into governed knowledge, architecture, implementation guidance, and technical continuity.

## Registration Rule

No new deity or king codename may enter canonical use until a registry record identifies:

- historical attribution;
- semantic justification;
- hierarchy placement;
- scope boundary;
- collision analysis;
- migration impact.
