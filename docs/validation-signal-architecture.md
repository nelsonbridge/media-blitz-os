# Validation Signal Architecture

## Purpose

The Validation Signal Architecture distinguishes genuine evolution from drift.

Most systems measure change. This system must measure whether change represents improved capability, stronger reality alignment, and preserved insight.

## Governing Question

Has the system become a better model of reality, or merely a different one?

## Change Classes

### Evolution

Capability increases without materially degrading coherence, evidence strength, transferability, or decision quality.

### Drift

The model changes without corresponding improvement in measurable capability.

### Regression

Capability decreases. Regression should trigger rollback, review, or quarantine.

### Mutation

A new capability appears that does not yet fit the taxonomy. Mutation requires investigation before classification.

## Validation Vector

Each candidate change may be evaluated across the following dimensions:

- Reality Alignment
- Predictive Gain
- Explanatory Gain
- Transfer Gain
- Coherence Gain
- Evidence Gain
- Decision Gain
- Generative Gain
- Conservation of Insight

## Conservation of Insight

A model demonstrates genuine evolution if it subsumes its predecessors, improves measurable performance against reality, and expands its ability to generate successful predictions or transferable architectures.

If a new model merely replaces prior understanding without preserving its valid explanatory power, it remains drift until validated.

## Validation Loops

### Learning Loop

Observation → Signal → Pattern → Principle → Framework → System

### Validation Loop

Prediction → Reality → Measurement → Calibration → Confidence Update

### Meta-Learning Loop

Taxonomy Gap → Structural Review → Classification Update → Retrospective Reclassification

## Promotion Rules

No observation becomes canonical merely because it is interesting.

Promotion requires one or more of:

- repeated evidence
- successful prediction
- cross-domain transfer
- improved explanatory power
- improved decision quality
- durable coherence with prior validated insights

## Drift Handling

Suspected drift is not discarded immediately.

It is placed in the Drift Review Queue with:

- trigger
- affected asset
- proposed change
- suspected drift reason
- validation needed
- review outcome

## Ancestry Requirement

Every mature idea should retain traceable ancestry:

Observation → Signal → Pattern → Principle → Framework → Publication/System

Ancestry allows the system to test whether later abstractions still preserve the evidence that originally justified them.
