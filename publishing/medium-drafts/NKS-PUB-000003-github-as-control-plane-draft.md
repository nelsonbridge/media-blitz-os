# GitHub Is the Control Plane for This Knowledge System

## Subtitle

Version control is not only for code. It can govern knowledge, execution, and continuity.

## Draft

A complex knowledge system needs a control plane.

A chat thread is not enough.

A folder is not enough.

A collection of drafts is not enough.

Those surfaces may hold work, but they do not automatically govern the work. They do not consistently preserve execution rules, state, schemas, queue logic, synchronization status, or the relationships between artifacts.

That is why GitHub matters in the Nelson Knowledge System.

Not as a code repository.

As the control plane.

## The Workspace Is Not the Control Plane

Google Drive is an excellent editorial workspace.

It is useful for documents, drafts, sheets, folders, visual planning, and human-readable collaboration.

But a workspace and a control plane are not the same thing.

The workspace contains the work.

The control plane governs how the work is created, tracked, verified, evolved, and recovered.

That distinction became necessary because the system needed continuity across sessions and resilience when one tool surface became unavailable.

If Drive is available, work can be written there.

If Drive is unavailable, work still has to continue.

GitHub provides that continuity.

## What GitHub Holds

In this system, GitHub holds the machine behind the work.

It stores:

- governance documents,
- operating rules,
- execution protocols,
- source records,
- artifact specifications,
- templates,
- publication packages,
- corpus indexes,
- relationship maps,
- synchronization ledgers,
- and master project state.

That means GitHub is not merely a backup.

It is the durable operating record.

## Why This Matters for AI-Assisted Work

AI-assisted work breaks down when the system has no stable state.

Without state, every session becomes reconstruction.

What was completed?
What is next?
Which rule governs execution?
What should be synchronized?
Which artifacts are canonical?
Which drafts are public-ready?
What is blocked?

If those answers live only in conversation, the system remains fragile.

A control plane changes that.

The assistant can consult a persistent structure, update it, and continue from the last verified state instead of rebuilding context from memory.

## Version Control for Knowledge

Version control is usually associated with software.

But the deeper idea is broader than code.

Version control preserves change history.

It makes the evolution of a system visible.

It allows governance to become durable.

It gives artifacts stable paths.

It makes updates auditable.

Those capabilities are just as useful for a knowledge system as they are for a software system.

## The Fallback Pattern

The practical pattern is simple:

```text
Preferred Workspace Available
      ↓
Write to Drive
      ↓
Update GitHub State

Preferred Workspace Unavailable
      ↓
Write to GitHub
      ↓
Mark Ready to Sync
      ↓
Synchronize Later
```

The system does not stop because one surface is unavailable.

It continues through the control plane.

## The Operating Principle

Separate the workspace from the control plane.

The workspace is where humans comfortably work.

The control plane is where the system preserves its rules, state, history, and recovery path.

For the Nelson Knowledge System, Drive can contain the work.

GitHub contains the machine that governs the work.

That distinction is what makes the system resilient.
